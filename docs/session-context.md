# Session Context Strategy

这是“会话系统与用户上下文”阶段的下一步：让 session 历史不仅能被模型使用，还能被稳定地控制。

当前项目采用的是“窗口式上下文注入”：

- 只读取最近 N 条 session 消息
- 再按总字符预算裁剪
- 只把 `user`、`assistant`、`tool` 三类消息喂给模型
- `system` 消息继续保留在存储层，不直接进入会话历史

## 为什么先做窗口，而不是直接做摘要

原因有两个：

- 实现简单，便于快速验证多轮会话是否有效
- 后续接摘要时，不需要推翻当前结构，只需要在窗口之外补一层 summary

这一步的目标不是“最聪明”，而是“先可控”。

## 当前使用的两个参数

### `SESSION_HISTORY_MESSAGE_LIMIT`

控制一次最多读取多少条历史消息。

默认值：

```bash
SESSION_HISTORY_MESSAGE_LIMIT=8
```

适合当前阶段的原因：

- 能覆盖最近几轮对话
- 不会让短会话也带上太多无关历史

### `SESSION_HISTORY_CHAR_BUDGET`

控制最终传给模型的历史消息总字符数预算。

默认值：

```bash
SESSION_HISTORY_CHAR_BUDGET=4000
```

当前实现策略：

- 从最近消息开始向前回收
- 超出预算时，只截取最后一段内容
- 优先保留“最近、最相关”的上下文

## 当前实现的优点

- 配置简单，容易调参
- 对短会话足够好用
- 能避免 session 一长就无限膨胀
- 给后续“摘要 + 最近窗口”方案留好了位置

## 当前实现的限制

- 只按“最近”裁剪，不按语义相关性裁剪
- 超长消息会被直接截断，可能切断句子
- 还没有把旧历史压缩成 summary

这些限制在当前学习阶段是可以接受的，因为我们现在主要在建立工程骨架和治理意识。

## 推荐调参方式

如果你在本地做实验，可以按这个顺序调：

1. 先调 `SESSION_HISTORY_MESSAGE_LIMIT`
2. 再调 `SESSION_HISTORY_CHAR_BUDGET`
3. 观察：
   - 第二轮回答是否还能记住第一轮信息
   - 工具型对话是否会被旧内容干扰
   - 长网页总结后，后续追问是否还能保持稳定

## 下一步该怎么演进

这个阶段完成后，下一步推荐做“摘要式会话记忆”：

- 近期消息：直接保留
- 更早消息：压缩成一段 session summary
- 模型输入：`summary + recent window`

这样就能从“短会话可用”进化到“长会话可维护”。
